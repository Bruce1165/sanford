# 老鸭头五图主线需求基线（v1）

> 版本：v1  
> 状态：生效中（作为后续开发与验收唯一主线）  
> 更新日期：2026-04-19

## 1. 项目主线目标

本项目服务于投资团队 A 股选股工作，核心目标是：

1. 以团队内部定义的筛选器为核心，持续输出稳定、可复现、可解释的候选股票池。
2. 通过参数可调与历史回放能力，使机器筛选结果逐步逼近团队人工筛选标准。
3. 通过“老鸭头预筛池 + 五图持续跟踪”机制，为团队提供更聚焦的重点跟踪池。

## 2. 主线功能范围

## 3. 数据基础与双数据源约束

五图运行依赖两个数据源，缺一不可：

1. 股票数据库（18 个月历史 + 每日增量）
2. 老鸭头股票池数据库（含 `start_date/end_date/processed`）

后续所有开发、验证、回归，必须明确区分并同时验证这两个数据源链路。

## 4. CHECK 回放主逻辑（必须保持）

对老鸭头池记录执行按交易日补跑核查（不再依赖 `upload_time/end_date` 兜底）：

1. 先判定池记录是否为 A 股有效股票（代码合法、在 `stocks` 存在、未退市、非指数代码）。
2. 设定本次补跑结束日：`min(调度触发日最近交易日, A 股数据库 latest_db_trade_date)`。
3. 设定本次补跑起始日（二选一）：
4. 若存在 `last_screened_date`，起始日 = `last_screened_date` 的下一交易日。
5. 否则起始日 = `start_date` 对应的交易日（若非交易日则顺延到下一交易日）。
6. 若起始日 > 补跑结束日，视为已最新，跳过该池记录。
7. 在 `[起始日, 补跑结束日]` 的每个交易日运行五图筛选器 CHECK，命中写入结果池。
8. 执行完成后推进 `processed` 与 `last_screened_date`。

触发补充说明：

1. 自动/默认手动触发前，先判断是否存在“需要补跑”的池记录。
2. 若全部池记录 `last_screened_date >= latest_db_trade_date`，则当日不触发补跑任务。

说明：该逻辑是业务闭环核心，不得在未评审前随意改写。

## 5. 去重规则（强制）

重复定义为三个字段同时重合：

- `stock_code`
- `screener_id`
- `screen_date`

即：`股票 + 筛选器 + 日期` 三键相同，视为重复，必须拦截。

## 6. 参数管理原则（强制）

1. 参数调整必须可追溯（参数版本、变更时间、变更说明）。
2. 同一参数快照必须可在两个数据源链路复现。
3. 参数变更后必须通过回放样本验证后才能进入发布流程。
4. 不允许“只在一侧生效”的参数状态长期存在。

## 7. 当前关键风险

1. 五图逻辑仅完成初步验证，尚未完成完整闭环验证。
2. 参数在双数据源链路中的生效一致性曾出现回归风险。
3. UI 迭代曾导致文件破损与功能回滚，后续必须限制改动范围。

## 8. 冻结与改动边界

为降低回归风险，执行以下边界：

1. 冻结 `Data Health` 区域与主布局骨架。
2. 五图迭代仅允许改动五图相关 API、脚本、右侧展示模块。
3. 禁止大范围全局样式改造。
4. 禁止改动历史归档目录中的文件。

## 9. 验收总标准（主线）

1. 同输入同参数快照，输出稳定一致。
2. 三键去重严格生效。
3. 参数调整可双链路复现与解释。
4. CHECK 回放闭环完整（可执行、可复跑、可追溯）。
5. 发布后可快速验证与回滚，避免长时间页面/发布调试消耗。
